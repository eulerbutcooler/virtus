Virtus

A transparent, participation-based community fulfillment platform. Architecture reference and decision log for every choice made in the codebase.

What is Virtus?

Virtus is a mutual support ecosystem where members both contribute to a shared pool and submit requests for essential items (laptops, mobility aids, educational tools, etc.). Requests are processed through a transparent, rule-based queue. When the pool accumulates enough funds, requests are automatically funded in order. Every step — from contribution to delivery to long-term impact — is tracked and publicly visible.

The key distinction from charity: every participant is simultaneously a contributor and a potential beneficiary. Access is earned through participation, not proof of hardship. This preserves dignity while creating a self-sustaining system.

🎯 Core Loop
  1. Member joins and contributes to the pool
  2. Member submits a request for an item
  3. Admin verifies and queues the request
  4. Worker funds requests as the pool fills
  5. Admin procures, ships, and verifies delivery
  6. Member reports impact; cycle continues
🏛️ Three Actor Types
  • member — individual contributor/beneficiary
  • admin — verifies requests, manages procurement
  • institution — corporate/NGO/government partner, contributes directly to the pool with tagging for ESG reporting

Decision — Two binaries

The API server and worker are separate binaries compiled from the same module. They share all internal packages but have independent lifecycles. This means you can scale, restart, and deploy them independently. The worker never touches HTTP; the API never runs tickers.

Decision — Domain-centric design

All repository interfaces live in internal/domain, not in the repository package. This means services depend on abstractions, not concrete implementations. Swapping PostgreSQL for another store or mocking for tests requires zero changes in service code.

Decision — Skip draft status

The domain defines a draft status but the API submits requests directly as submitted. This avoids a save-then-submit two-step flow in MVP. Draft exists in the schema so members can eventually save incomplete requests before submitting — the service and SQL already support it.

Decision — Worker funds, not the API

Funding is not triggered by a webhook or a direct API call — it is discovered by the polling worker. This makes it resilient: if a contribution's webhook fires while the queue is being evaluated, the next poll catches it. There is no race condition between "payment completed" and "request funded" events.

Auth & Security

Token Strategy

Two-token pattern: short-lived access tokens (15 minutes) and longer-lived refresh tokens (7 days). Access tokens are stateless JWTs validated by the middleware without a DB hit. Refresh tokens are stored in Redis so they can be individually invalidated — logging out one device doesn't affect others.

JWT Claims payload
{
  "uid":  "uuid-of-user",
  "role": "member",          // embedded so middleware avoids a DB call
  "aud":  "virtus:access",   // "virtus:refresh" for refresh tokens
  "jti":  "random-16-bytes", // used as Redis key suffix for RT invalidation
  "sub":  "uuid-of-user",
  "iat":  1234567890,
  "exp":  1234567890
}

Why audience claim?

Access and refresh tokens have different aud claims (virtus:access vs virtus:refresh). The parser enforces this, making it impossible to use a refresh token where an access token is expected and vice versa.

Redis key pattern

Each refresh token is stored at rt:{user_id}:{jti}. On logout, only that specific JTI is deleted. On refresh, the old JTI is deleted and a new pair is issued. This allows per-session logout without forcing all sessions to expire.

Role-Based Access

Roles are embedded in the JWT so RequireRole(domain.RoleAdmin) middleware needs no database call. Three roles:

member

Default role. Can submit requests, contribute to the pool, view their own data, record impact.

institution

Like member but can also register an institution profile and make tagged institutional contributions.

admin

Can verify/reject requests, begin fulfillments, ship and verify deliveries, verify institutions. All admin routes are grouped under /admin/ and gated by RequireRole(RoleAdmin).

Decision — Passwords with bcrypt

bcrypt is used via pkg/crypto with Go's golang.org/x/crypto/bcrypt. The cost factor defaults to 12 (bcrypt default). Password hashes are never returned in JSON responses — the User struct uses json:"-" on PasswordHash.

Step-by-step

  1. POST /contributions — Member specifies amount + currency. The API calls ContributionService.Initiate, which creates a Stripe PaymentIntent and a pending Contribution row (with stripe_pi_id stored). The client_secret is returned to the frontend.
  2. Frontend confirms payment — The browser uses Stripe.js + the client secret to collect the card and confirm. This never touches the Virtus backend directly.
  3. POST /webhooks/stripe — Stripe fires a signed webhook. The handler reads the raw body, verifies the Stripe-Signature header using the webhook secret, then routes by event type.
  4. payment_intent.succeeded → ContributionService.Complete runs inside a DB transaction: marks the contribution completed and CreditPool atomically. If either step fails, both roll back.
  5. payment_intent.payment_failed / canceled → ContributionService.Fail marks the contribution failed. The pool is untouched.
  6. Procurement — When an admin sets actual_cost on a fulfillment, the pool is debited by that amount. This is separate from estimated cost (actual may differ).
Decision — Idempotent webhook handler

Stripe can deliver the same webhook event more than once (at-least-once delivery). ContributionService.Complete checks if the contribution is already completed and returns nil without re-crediting the pool. The webhook always returns HTTP 200 for processed events so Stripe doesn't retry unnecessarily.

Decision — PaymentProvider interface

ContributionService depends on a PaymentProvider interface, not on *stripe.Provider directly. This means the service is testable without hitting Stripe, and swapping to another payment processor (Dodo Payments, Braintree, etc.) requires only a new implementation of two methods: CreateIntent and (separately) ParseWebhook.

Decision — Webhook is a public route

The Stripe webhook at POST /webhooks/stripe is registered outside the RequireAuth middleware group. Stripe cannot send a Bearer token. Security is provided by signature verification (webhook.ConstructEvent) using the webhook secret — which is equivalent in strength to a shared secret auth scheme.

Queue System

The queue is the transparency mechanism of the platform. Every verified request gets a QueueEntry with a position number. Members can see exactly where they stand and estimate when they'll be funded.

Queue Rules

  • FIFO ordering — Positions are assigned by insertion order (time of verification). The queue started with a priority score system (see migration 10) but this was dropped in favour of pure FIFO to prevent gaming the score.
  • One-to-one — A request can appear in the queue exactly once (UNIQUE constraint on request_id).
  • Dequeued on funding — When the worker marks a request funded, it is removed from the queue. The admin then begins procurement separately.
  • Dequeued on rejection — If an admin rejects a queued request, it is removed (best-effort — the request status update always succeeds even if the dequeue call fails).

Funding Progress Field

Each QueueEntry has a funding_progress float. Currently written to 0.0 on creation and set to 100.0 (implicitly, by removal) on funding. This field is the hook for a future partial-funding display — e.g., showing the pool is "62% of the way" to covering a request.

Decision — Dropped priority score (migration 10)

Migration 5 added a priority_score column and index. Migration 10 drops both. The score was intended to weight urgency, wait time, and contribution history — but this creates perverse incentives (members game urgency levels) and undermines the "fair queue" promise. Pure FIFO is simpler, more legible, and harder to manipulate.

Worker

A separate binary (cmd/worker) that runs three periodic background tasks. No external job queue — just time.Ticker goroutines with graceful shutdown via context cancellation.

TaskIntervalWhat it does
fund_queue 30 seconds Reads pool balance, walks queue FIFO, calls RequestService.MarkFunded for each affordable request. Stops at the first it can't cover. Logs each funding event.
impact_reminder 24 hours Joins deliveries → fulfillments → requests. Finds verified deliveries >30 days old missing a 30/90/180-day impact record. Logs each gap for admin follow-up.
delivery_watchdog 24 hours Finds deliveries in in_transit with updated_at >14 days ago. Logs a warning per stale delivery. Does NOT auto-fail — human decision required.

Task Runner Pattern

// runPeriodic in cmd/worker/main.go
func runPeriodic(ctx, interval, name, fn) {
    run()              // run immediately on start
    ticker := Ticker(interval)
    for {
        select {
        case <-ctx.Done(): return
        case <-ticker.C:  run()
        }
    }
}

Each task is isolated: an error in one task logs and is skipped; the ticker fires again on the next interval. The worker never crashes on a task failure.

Decision — No asynq / River / job queue

We considered asynq (Redis-backed) and River (Postgres-backed). Both add significant complexity for what are essentially three polling loops. Per the simplicity principle: the current tasks don't need distributed scheduling, retry queues, or job dashboards. If the platform grows to need guaranteed-delivery async jobs (e.g., email notifications), asynq can be introduced at that point without structural changes to the rest of the codebase.

Decision — Worker uses pgxpool directly for watchdog queries

The impact_reminder and delivery_watchdog tasks run queries that join multiple tables in ways not covered by the existing domain.DeliveryRepository interface. Rather than extending the interface (which would require implementing it in the repo, adding SQL, regenerating sqlc), these tasks hold a *pgxpool.Pool and run raw queries. This is acceptable because they are read-only diagnostic queries, not business logic.

Database Design

Schema overview

Ten tables, all using UUID PRIMARY KEY DEFAULT gen_random_uuid(). Timestamps are TIMESTAMPTZ (timezone-aware). Monetary amounts are NUMERIC(15,2) — never FLOAT, which cannot represent decimal values exactly.

TableNotable constraints
users email UNIQUE, role is a Postgres ENUM
pools Single row seeded at startup. balance CHECK (balance >= 0) prevents overdraft at the DB level.
contributions stripe_pi_id UNIQUE prevents duplicate webhook processing. amount CHECK (amount > 0).
requests status and urgency are Postgres ENUMs. estimated_cost CHECK > 0.
queue_entries request_id UNIQUE (one queue slot per request). ON DELETE CASCADE — deleting a request clears its queue entry.
fulfillments request_id UNIQUE. ON DELETE RESTRICT — can't delete a request that has a fulfillment in progress.
deliveries verified_at, delivered_at, verified_by are nullable; set only when verified.
impact_records No uniqueness on (delivery_id, interval_label) — multiple records per interval are allowed (the worker checks for existence in Go, not at the DB level).
institutions user_id UNIQUE — one institution per user account. verified flag set by admin.
institutional_contributions Separate from contributions to carry institutional metadata (category_tag, region_tag, notes) for ESG reporting.

sqlc

All queries are written by hand in internal/repository/postgres/queries/*.sql and compiled to type-safe Go by sqlc. The generated code lives in internal/repository/postgres/db/ and is never edited manually. Repository implementations in internal/repository/postgres/ wrap the generated code, translating between dbgen.* types and domain.* types.

Decision — NUMERIC not FLOAT for money

IEEE 754 floating point cannot represent 0.10 exactly. All monetary columns use NUMERIC(15,2). Go maps this to float64 via pgx's numeric codec, which is sufficient for display and comparison. A future hardening step would use a decimal.Decimal type in Go as well.

Decision — ON DELETE RESTRICT on financial rows

Contributions and fulfillments use ON DELETE RESTRICT on their foreign keys. You cannot delete a user who has contributed, or a request that has been procured. This preserves the audit trail. The queue uses ON DELETE CASCADE because a queue entry has no independent value once its request is gone.

Decision Log

Every significant design choice, and the reasoning behind it.

Architecture

Pure domain layer

All repository contracts (UserRepository, PoolRepository, etc.) are interfaces defined in internal/domain, not in the repository package. This is a deliberate inversion: the business logic defines what it needs; the infrastructure provides it. Services can be tested by injecting mock repos with no framework or build tags required.

Sentinel errors, not error codes

Domain errors are typed sentinels: ErrNotFound, ErrConflict, etc. Service errors wrap them with fmt.Errorf("...%w", domain.ErrNotFound). Handlers use errors.Is to map to HTTP status. This means error messages can be human-readable while still being machine-checkable — no stringly-typed error codes, no HTTP-awareness in the service layer.

No global state

The Stripe SDK sets a global key (stripelib.Key = secretKey) in pkg/stripe.New. This is the only global mutation in the codebase and is a concession to the Stripe SDK's design. Everything else — DB pool, Redis client, all services — is passed explicitly via constructor injection.

Service Layer

Services own transactions, not repos

ContributionService.Complete begins a pgxpool.Pool.Begin transaction, then creates a dbgen.Queries scoped to that transaction (dbgen.New(tx)). This pattern lets services compose atomic operations across multiple repo calls without leaking transaction knowledge into the repository interface. The repo interface has no WithTx method.

Verify → Queued in one admin call

RequestService.Verify does three things in sequence: update status to verified, enqueue, update status to queued. These are deliberately not wrapped in a transaction. If the enqueue fails, the request sits at verified and can be re-processed. A rollback would lose the admin's verification decision. Partial failure here is recoverable; a rollback would create more confusion.

MarkFunded does not debit the pool

When the worker marks a request funded, the pool balance is not debited. The debit happens later in FulfillmentService.UpdateDetails when the admin records the actual cost (which may differ from the estimated cost). This means between "funded" and "actual cost set", the balance appears inflated. For MVP with a small admin team this is acceptable — the admin acts quickly. A future hardening step would reserve the estimated cost at funding time.

HTTP Layer

Versioned API under /api/v1/

All routes are under /api/v1/ except the health check and Stripe webhook. The webhook is at the root because Stripe's dashboard expects a clean URL and you can't add a path prefix to webhook deliveries after the fact. The health check is at /healthz for standard k8s/load-balancer compatibility.

DecodeAndValidate disallows unknown fields

json.NewDecoder(r.Body).DisallowUnknownFields() is set. This prevents clients from sending fields that silently do nothing — a common source of confusion when a client typos a field name. It also provides a forcing function: the API contract is explicit about what fields are accepted.

Ownership checked in handlers, not services

Handlers call canAccess(ctx, req.UserID) before allowing reads or mutations on owned resources. Services don't do ownership checks — they trust the handler. This keeps services reusable (e.g., the worker calling MarkFunded doesn't need to impersonate an admin) and keeps authorization logic in one obvious place.

Database

pgcrypto for UUID generation

Migration 1 runs CREATE EXTENSION IF NOT EXISTS "pgcrypto" so that gen_random_uuid() is available. UUIDs are generated at the database level, not in Go. This means even direct SQL inserts (e.g., seed scripts) get proper UUIDs without any application-layer involvement.

Postgres ENUMs for status fields

Request status, fulfillment status, urgency level, contribution status, delivery status, and institution type are all Postgres ENUM types. This enforces validity at the DB level — a bug can't write "PENDING" (wrong case) or "in-transit" (wrong separator). sqlc generates matching Go types for each.
